- 静的ルーティング
```
localhost:3000/users/0 → app/users/0/page.tsx
localhost:3000/users/1 → app/users/1/page.tsx
localhost:3000/users/2 →app/users/2/page.tsx
```
↓

- ダイナミックルーティング
- [userId]というようにパラメータ化して１つのページで複数のユーザーの情報を切り替えることができる
```
app/users/[userId]/page.tsx
```
- 実際に表示させるには、
```
app/users/[userId]/page.tsx

page.tsxにて

interface Props {
  params:{ userId: string };
}

export default function UserPage(props: Props) {
  return (
      <>
           <h1 className="text”-lg border-b pb-1 mb-1”>User Page{ props.params.userId }</h1>
      </>
  )
}
```
- link
- ページ全体を再読み込みするのではなく最小限の読み込みで表示できる機能
```
<Link href="/users/0">User 0<Link>
```

- エラーメッセージ
- URLがなかった場合：基本は404エラーを出すが、localhost:3000/users/1000と存在しない場合、予期せぬruntimeエラーが出てしまう

- 早期リターンする
```
if (user === undefined) {
　notFound();
}
→notFoundページが出る
```
- このページをカスタマイズしたい場合
not-found.tsxを作成し編集する

- Server Actions
Next.js 13/14以降の目玉機能です。 「フォームから送信されたデータを、APIを作らずに直接PrismaでDBに保存する」仕組み。これが分かれば、値を取得する だけでなく 値を保存・更新・削除する ことができるようになる。
- データの取得タイミング (Server Components)
Next.jsでは、page.tsx の中で直接 prisma.user.findMany() のようにDBを叩ける。これを Server Components と呼ぶ。
- 認証 (NextAuth.js / Auth.js)
フルスタックアプリには「ログイン機能」がほぼ必須。 「この投稿は、ログインしているこのユーザーのものか？」という判定をPrismaのデータと照合するスキルが必要。
