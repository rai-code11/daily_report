## わかったこと

onClick は、「クリックされた時に、この関数を実行してね」という指示です。 Reactはクリック時に自動的にイベントオブジェクト（e）を最初の引数として渡してくれるので、自分でわざわざ (e) => ... と書いてリレーする必要がない

Promiseとawait
```js
async function main() {
  A(); // 同期
  B(); // 同期

  // Cをスタートさせるが、まだ「await（待機）」はしない！
  // 「約束手形（Promise）」だけもらってポケットに入れる
  const promiseC = C(); 

  D(); // Cが裏で動いている間に、Dが実行される！

  //  ここで初めて「C終わった？」と確認して合流する
  await promiseC;
}
```
実際の流れ:
1. A 実行
2. B 実行
3. C 開始（裏へ回す）
4. D 実行（Cを待たずに動く！）
5. C 完了待ち（合流）


結論
* await C() と書いた瞬間、その下の行（D）は全て「Cが終わるまでお預け」になる
* Dを先に動かしたいなら、await は Dの後 に書く必要がある


1. 直列処理（ロスが大きい）
    * 「処理開始 → await（ここで止まって待つ） → 終わったら次の処理」
    * 丁寧だけど、待ち時間がもったいない。
2. 並列処理（効率が良い）
    * 「処理開始（Promise作成） → 待たずに次へ！（裏で走らせる） → 最後に await で結果を受け取る」
    * 裏で煮込んでいる間に、別の野菜を切るようなもの。これがやりたかったことです！

- コードで見る「ロス」と「効率」の差
- データの取得にそれぞれ2秒かかるとする
- パターンA：ロスが大きい（直列）

```JavaScript
// 1. ユーザーデータを取ってくる（2秒待つ...）
const user = await getUser(); 

// 2. その後で、商品データを取ってくる（さらに2秒待つ...）
const items = await getItems();

// 合計：4秒かかる 
- パターンB：効率が良い（並列）
あなたが気づいた、「Promiseを設置して裏でやる」パターンです。
```JavaScript


// 1. 「取ってきてね」と依頼だけして、Promise（引換券）をもらう
// （ここでは await しない！＝止まらない）
const userPromise = getUser();
const itemsPromise = getItems();

// この時点では、裏で2つの通信が「同時に」走っています！

// 2. 最後に結果を開封する
const user = await userPromise;
const items = await itemsPromise;

// 合計：2秒ちょっとで終わる！ 
// （同時にスタートしたので、遅い方が終われば両方終わっている）
```
実際の現場では…
この「複数のPromiseを同時に走らせて、全部終わるのを待つ」という処理はあまりによく使われるため、Promise.all という専用の必殺技が用意されている
```JavaScript


// 2つ同時にヨーイドン！
const [user, items] = await Promise.all([
  getUser(), 
  getItems()
]);
```
これを使うと、配列で渡した処理が全部並列で走り、全部終わったタイミングで結果をまとめて返してくれる
