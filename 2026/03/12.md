## わかったこと

- 種類処理のタイミング特徴Client Side (CSR)ブラウザに届いてからユーザーのPC上でJSを動かして画面を作る。
- Server Side (SSR)リクエストが来た瞬間サーバーがその場でHTMLを組み立ててから送る。Static Generation (SSG)

- useEffectを一言で言うなら、**「レンダリング（画面の書き換え）が終わった後に、画面表示以外の『余計な仕事』をさせるためのフック」
-「副作用」って具体的に何？
- コンポーネントの本来の仕事は「UIを計算して出すこと」です。それ以外の、以下のような処理がuseEffectの出番。
* API通信: サーバーからデータを取ってくる
* DOMの直接操作: グラフライブラリを動かしたり、手動でフォーカスを当てたりする
* タイマー設置: setTimeout や setInterval
* イベント監視: ブラウザのスクロールやリサイズを検知する

- 第2引数の指定	実行されるタイミング	よくある用途
- なし (useEffect(() => {}))	レンダリングのたびに毎回	基本的にはあまり使わない
- 空の配列 ([])	最初の1回だけ（マウント時）	初期データの取得、外部ライブラリの起動
- 変数あり ([count])	その変数が変わった時だけ	特定の値に連動した処理（検索窓の入力など）
- 「〜したとき（Click, Submit）」 → 普通の関数
- 「〜な状態になったとき（Mount, Update）」 → useEffect


- ボタンを押してデータを取得する場合（イベント駆動）
- これは useEffect を使いません。関数のなかで直接実行

```JavaScript
function UserProfile() {
  const [data, setData] = useState(null);

  // ボタンが押された時に実行される「ただの関数」
  const handleClick = async () => {
    const response = await fetch('https://api.example.com/user');
    const result = await response.json();
    setData(result); // データを取得してStateに入れる
  };

  return (
    <button onClick={handleClick}>データを取得する</button>
  );
}
```

- 画面が表示された瞬間に取得する場合（状態駆動）

```
function UserProfile() {
  const [data, setData] = useState(null);

  // レンダリングが終わった後に「自動で」実行される
  useEffect(() => {
    const fetchData = async () => {
      const response = await fetch('https://api.example.com/user');
      const result = await response.json();
      setData(result);
    };

    fetchData();
  }, []); // 空の配列 = 最初の1回だけ実行

  return (
    <div>{data ? data.name : "読み込み中..."}</div>
  );
}
```
