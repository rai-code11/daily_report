## わかったこと

## 1. なぜ `params` は Promise になったのか？

* **以前（同期）:** ページを開いた瞬間、すでにデータが揃っている前提。
* **現在（非同期）:** 必要なタイミングでデータを取りに行く（StreamingやPPRの最適化のため）。

****

> **イメージ：** > 昔は「中身が詰まった状態で届く箱」だったが、今は**「引換券（Promise）」**が届くイメージ。そのままでは中身が使えないから、`await` という手続きをして本物のデータを受け取る必要がある。

---

## 2. 実装の合言葉は「まず `await params`」
サーバーコンポーネントでURLの変数（`todoId`など）を使いたい時は、以下のステップが必須になる

1.  **型の定義:** `params: Promise<{ ... }>` と定義する。
2.  **待機:** `const { todoId } = await params;` で中身を取り出す。
3.  **利用:** 取り出した後の `todoId` を使って、DB（Prisma）へ検索をかける。

```typescript
// 1. 型は Promise
interface Props {
  params: Promise<{ todoId: string }>;
}

const TodoPage = async ({ params }: Props) => {
  // 2. まずは await して中身（箱）を開ける
  const { todoId } = await params;
  
  // 3. 確定した値を使って処理をする
  const todo = await prisma.todo.findUnique({ ... });
}
```

---

## 3. 型定義は「自分で書かない」

* **手書きのリスク:** DBのテーブル（Schema）を変えたとき、コード側の型定義も手動で直す必要があり、ミス（ズレ）が起きやすい。
* **Prisma Clientの活用:** `@prisma/client` から型をインポートすれば、DBの構造とコード上の型が常に**100%一致**する。
