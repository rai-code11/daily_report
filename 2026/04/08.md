## わかったこと

## 1. REST APIの設計（URLとメソッド）
同じURLでも、**HTTPメソッド（動詞）**を組み合わせることで、異なる操作を実現する。

| メソッド | URL (エンドポイント) | 役割 |
| :--- | :--- | :--- |
| **GET** | `/todos` | データ一覧を取得する |
| **POST** | `/todos` | 新しいデータを作成・追加する |
| **DELETE** | `/todos/:id` | 特定のIDのデータを削除する |

- **ポイント**: URLを機能ごとに増やすのではなく、リソース（名詞）を固定してメソッドで制御するのがモダンな設計。

## 2. データのパッキング（シリアライズ）
JavaScriptのオブジェクトはそのままでは通信できないため、変換作業が必要になる。

- **フロントエンドからの送信 (`POST`)**:
  - `JSON.stringify({ title: input })` を使い、オブジェクトを**JSON文字列**に変換して送る。
  - `headers: { "Content-Type": "application/json" }` で「これはJSONです」と宣言する。
- **バックエンドからの返却 (Hono)**:
  - `return c.json(data)` と書くことで、Honoが自動的に文字列への変換とヘッダー付与を行ってくれる。



## 3. Honoの `Context (c)` の役割
ハンドラーの引数 `c` は、リクエストとレスポンスを繋ぐ司令塔。

- **`c.req`**: フロントから届いた情報（パラメータ、クエリ、JSONボディ）を取り出す。
- **`c.json()`**: フロントへ返すレスポンス（データ、ステータスコード）を生成する。

## 4. 通信のキャッチボールの流れ
API通信は、インターネットを介した「関数の呼び出し」と同じ構造。

1. **呼出し (Front)**: `fetch` でバックエンドの関数（API）を叩く。
2. **実行 (Back)**: Honoがリクエストを受け取り、PrismaなどでDB操作をして `return c.json()` で値を返す。
3. **受取 (Front)**: 届いた `res` を `await res.json()` でJavaScriptオブジェクトに戻して利用する。



## 5. あえて「生」で書く意義（fetch vs Axios）
Axiosなどのライブラリを使わず、標準の `fetch` を使うことで通信の裏側が可視化される。
- `Content-Type` の指定
- `JSON.stringify` によるシリアライズ
- `res.json()` によるデシリアライズ

これらを一つずつ手書きすることで、**「今、データがどの形（オブジェクトなのか文字列なのか）になっているか」**を正確に把握でき、デバッグ能力の向上に繋がる。
