## わかったこと

## 1. システム構成の全体像
これまでのフルスタック（Django等）とは異なり、役割を完全に分担させる構成。

* **バックエンド (Hono + Prisma + PostgreSQL)**
    * 役割：データベース（倉庫）の管理と、データの出し入れ。
    * 特徴：画面を持たず、**JSON**という「生データ」を吐き出すことに特化する。
* **フロントエンド (Next.js)**
    * 役割：ユーザーが見る画面（UI）の作成。
    * 特徴：バックエンドからJSONを受け取り、Reactで綺麗に表示（盛り付け）する。

## 2. 通信の仕組み：REST API
二つの独立した世界をつなぐ「共通のルール」。

| 要素 | 役割 | 具体例 |
| :--- | :--- | :--- |
| **窓口 (URL)** | どこに対して指示を出すか | `http://localhost:3003/todos` |
| **動詞 (Method)** | 何をしてほしいか（指示の内容） | GET(取得), POST(作成), DELETE(削除) |
| **共通言語 (JSON)** | データのやり取りに使う形式 | `{"id": 1, "title": "宿題"}` |

## 3. 重要コンセプト：`fetch` 関数
フロントエンドからバックエンドへ「指示」を出すための唯一の手段。

* **指示の出し方（例：削除）**
    * 「どの窓口に（URL）」「どの動詞で（DELETE）」送るかを明示する必要がある。
    * これがないと、バックエンドは誰からのどんな指示か理解できない。
* **非同期処理 (`await`)**
    * 通信には時間がかかるため、返事を待ってから次の処理（画面更新など）を行う。

## 4. 本日の気づき
* フルスタックは「全部入り」で便利だが、分離させることで**「通信の裏側」**が可視化される。
* バックエンドは「適切な窓口で、適切なJSONを返す機械」と捉えると設計がシンプルになる。
* 独立したサーバー同士が**JSON**という共通言語で会話することで、一つのアプリが成立している。

## 感じたこと
- 今日からHonoとNext.jsを使ってフロントとバックエンドを分けた開発に挑戦してるけど、ぜんぜん違う。

今まではDjangoとかNext.jsのフルスタックだったけど、あえて切り離すことでWebの仕組みがよりわかるようになった。

バックエンドでAPIを用意して、DBからデータを取ってきてJSONで吐き出す。

フロントエンドは、APIにfetchで指示を出しに行く。返ってきたJSONを飲み込んで、Reactで綺麗に表示させる。

削除ボタンひとつとっても、フロントからfetchでDELETEっていう指示を添えて、正しいAPIを叩かないとバックエンドの処理は動いてくれない。

この独立したサーバーが共通言語のJSONを使ってやりとりしてるっていう感覚が掴めると、一気にWeb開発の解像度が上がる気がする！
